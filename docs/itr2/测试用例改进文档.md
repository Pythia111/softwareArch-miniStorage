## 3. 改进方案

### 3.1 核心原则：不硬编码，真实测试功能

硬编码测试的问题：只验证"这个输入得出这个输出"，无法检测**逻辑是否正确**。例如：

```java
// ❌ 硬编码：只能验证这一个特定输入
assertEquals("/usr/local", PathUtil.normalize("//usr///local"));

// ✅ 真实测试：验证方法间逻辑一致性，换输入也能发现 bug
String normalized = PathUtil.normalize(raw);
String[] parts = PathUtil.split(normalized);
if (parts.length > 0) {
    assertEquals(PathUtil.getBaseName(normalized), parts[parts.length - 1]);
}
```

### 3.2 新增测试维度

| 维度 | 说明 | 示例 |
|---|---|---|
| **语义陷阱** | 输入中含合法但易混淆的路径名 | `.hidden` ≠ `.`，`...` ≠ `..`，`b.c.d` 不拆分 |
| **组合性验证** | 多个方法组合使用时逻辑一致 | `getParentPath() + "/" + getBaseName()` 还原原路径 |
| **往返测试** | 正向 + 逆向操作结果一致 | `split()` 再 `join` 能还原 `normalize()` 的输出 |
| **链式回退** | 连续调用同一方法的边界行为 | `getParentPath()` 逐层回退到根再到 null |
| **parse vs 直接调用** | 聚合方法与手动分步调用结果一致 | `parse(raw)` 各字段 = 分别调用 normalize/split/getParentPath/getBaseName |

## 7. 语义陷阱清单

以下输入在路径规范化中容易被错误处理，是测试重点：

| 陷阱输入 | 错误理解 | 正确处理 | 测试用例 |
|---|---|---|---|
| `.hidden` | 等同于 `.`（当前目录） | 是合法文件名，保留原样 | `normalize_hiddenFileNameNotTreatedAsDot` |
| `...` | 等同于 `..`（父目录） | 是合法文件名，保留原样 | `normalize_tripleDotsIsRegularName` |
| `b.c.d` | 包含扩展名分隔符，应拆分 | 是单个路径组件，不拆分 | `split_nameWithDotsNotSplit` |
| `/a/.../b/.hidden/../c` | 多个陷阱混合 | `...` 和 `.hidden` 保留，`..` 正常回退 | `normalize_dotAndDotDotMixedWithRealNames` |
